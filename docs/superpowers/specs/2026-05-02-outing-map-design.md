# 外出地图系统 MVP 设计

> **状态**: Approved for Spec Review  
> **作者**: User + Codex  
> **最后更新**: 2026-05-12  
> **范围**: 外出地图 / 地点选择 / 出发准备 / 探索结算 / 庇护所库存 / 存档持久化

---

## 1. 目标

实现一个独立的 2D 外出地图场景，作为避难所内外出行为的主入口。

第一版目标不是做开放地图探索，而是完成一条稳定可扩展的外出主循环：

- 从避难所进入外出地图场景
- 在可拖拽、可缩放的地图上点击地点
- 查看地点详情
- 打开出发准备面板
- 从庇护所总库存汇总中选择携带物资
- 受 12 格外出携带栏限制（每个物品占 1 格，非武器可堆叠）
- 确认出发后推进时间并执行探索结算
- 获得随机物资，按地点偏向提高对应物资概率
- 解锁相邻更远地点
- 外出获得物资写回庇护所对应柜子或外出带回包
- 地图发现与探索次数进入全局存档
- 显示结果并返回进入外出地图前的避难所场景

后续该系统需要能够接入 AI 大模型生成探索叙事，因此本设计要求“规则结算”和“叙事生成”彻底分层。地点资源中的 `ai_exploration_rule` 只给 AI 使用，不作为普通 UI 文本展示。

---

## 2. 核心体验

玩家进入外出场景后，看到一张可操作的 2D 地图。地图不是菜单列表，而是整个玩法中心。

核心体验应满足以下几点：

1. 地图本身可拖拽、可缩放，具有浏览和扩展探索范围的感觉
2. 地点通过点位标记表达，点击点位后弹出地点详情卡
3. 详情卡用于了解地点性质，而不是承担全部管理功能
4. 真正的出发配置在二级准备面板中完成
5. 探索结果带有随机性，但会被地点偏向和玩家准备行为修正
6. 外出的第一版成本只体现为“时间”和“携带栏占用/消耗品扣除”，不加入损坏、电量、耐久惩罚

---

## 3. 非目标

以下内容明确不纳入第一版 MVP：

- 开放式外部实景探索
- 外出中的实时遭遇事件玩法
- 机器人损坏、电量、耐久消耗
- 多角色编队外出
- 复杂路线规划
- 天气系统
- 复杂工具条件链
- AI 探索故事生成
- 大型结果剧情分支

这些内容以后可以建立在当前外出地图系统之上，但第一版不进入实现范围。

---

## 4. 场景形态

采用 **独立 2D 场景** 实现外出系统，而不是在避难所 3D 场景里叠加一个全屏 UI。

### 原因

- 地图拖拽、缩放、点位浏览更适合单独的 2D 场景
- 可以将外出系统与避难所内交互、角色控制、3D 面板逻辑解耦
- 后续增加地点卡、相邻解锁、雾区、路线线段等可视元素时更稳定
- 更适合承载探索结算和未来的 AI 探索叙事入口

### 场景切换流

- 避难所中触发“外出”交互
- 切换到 `OutingMap.tscn`
- 完成探索后返回避难所主场景

---

## 5. 交互总流程

### 5.1 进入地图

玩家从避难所进入外出地图场景。

### 5.2 浏览地图

玩家可以：

- 鼠标拖拽地图平移
- 鼠标滚轮缩放地图
- 点击地点点位
- 点击空白处关闭地点详情卡

### 5.3 查看地点详情

点击点位后弹出地点详情卡，显示：

- 地点名称
- 简短描述
- 风险等级
- 预计总耗时
- 大概率物资类型
- 是否可能发现更远地点
- `准备出发` 按钮

### 5.4 准备出发

点击 `准备出发` 后打开二级准备面板。

准备面板负责：

- 读取庇护所各柜子的汇总物资
- 选择本次携带物资
- 根据 12 格携带栏判断是否可继续加入
- 展示本次探索的预计耗时
- 确认出发

### 5.5 探索结算

确认出发后直接执行：

- 扣除外出携带物资；武器/工具/特殊物品返还原来源格，消耗品按规则消耗
- 按地点掉落池、地点偏向、随机权重结算带回物资
- 将带回物资优先写入对应柜子，放不下时进入外出带回包
- 记录本地点探索成功次数并解锁满足条件的外缘地点
- 生成结果数据并触发存档

### 5.6 结果展示

结果面板显示：

- 前往地点
- 总耗时
- 获得物资
- 解锁新地点
- 返回避难所

---

## 6. 地图 UI 结构

地图场景以“地图为主，卡片为辅，准备面板为二级弹层”为核心结构。

### 6.1 地图层

地图层负责：

- 地图底图展示
- 拖拽和平移
- 缩放
- 点位承载
- 相邻关系表现
- 已探索 / 未探索的可视状态

地图应是玩家进入场景后的视觉中心，不应被大量固定侧栏挤压。

### 6.2 地点详情卡

地点详情卡不是常驻右侧大栏，而是“选中地点后弹出的浮动信息卡”。

它的职责仅为：

- 展示地点信息
- 提供前往准备入口

它不承担复杂库存管理。

### 6.3 出发准备面板

准备面板是单独的二级弹层。

它承担：

- 庇护所汇总物资选择
- 12 格携带栏与堆叠数量显示
- 本次出发确认

### 6.4 结果面板

探索完成后，结果面板显示结构化结算结果，后续可扩展 AI 叙事文本区域。

## 7. 推荐场景节点结构

建议创建单独场景：`OutingMap.tscn`

```text
OutingMapRoot
├─ Background
├─ DragZoomMapRoot
│  ├─ MapImage
│  ├─ FogLayer
│  ├─ RouteLayer
│  ├─ MarkerLayer
│  └─ ShelterMarker
├─ LocationDetailCard
├─ ExpeditionPreparePanel
├─ ResultPanel
├─ TopBar
├─ CloseButton
├─ AnimationPlayer
├─ AudioStreamPlayer
└─ OutingMapController
```

### 节点职责说明

#### `DragZoomMapRoot`
负责：
- 鼠标拖拽平移
- 滚轮缩放
- 最小 / 最大缩放限制
- 地图边界限制，避免拖飞

#### `MapImage`
负责地图底图显示，不放业务逻辑。

#### `FogLayer`
负责未探索区域的遮罩或淡化表现。第一版可先做简化状态，不必实现复杂动态蒙版。

#### `RouteLayer`
负责显示地点之间的邻接关系、探索延伸方向或已建立联系。

#### `MarkerLayer`
负责所有地点点位的实例化与刷新。

#### `LocationDetailCard`
负责展示当前选中地点信息。

#### `ExpeditionPreparePanel`
负责携带物资配置与确认出发。

#### `ResultPanel`
负责展示探索结算结果。

#### `OutingMapController`
总控脚本，负责：
- 读取地图资源和地点资源
- 管理点位显示与状态
- 管理当前选中地点
- 打开/关闭详情卡与准备面板
- 调用探索结算逻辑
- 写回庇护所库存与地图发现进度
- 触发保存并处理返回避难所

## 8. 地点点位场景

建议每个地点点位做成单独场景：`OutingLocationMarker.tscn`

推荐结构：

```text
OutingLocationMarker
├─ Button
├─ Icon
├─ SelectedRing
├─ NewUnlockHint
└─ DangerBadge
```

### 点位状态

- 未解锁：不显示或不可见
- 已解锁未探索：正常显示
- 已探索：更亮或更完整
- 当前选中：显示高亮圈
- 新解锁：显示提示点或短时强调动画

## 9. 数据架构

外出地图系统必须数据驱动。

### 9.1 地点规则资源

当前实现使用 `levels/outing/location_rules/*.tres`。每个地点一个规则资源，运行时由 `levels/outing/outing_map_level_v3.gd` 读取。

#### 字段建议

##### 基础信息
- `location_id`
- `display_name`
- `description`
- `map_position`
- `start_unlocked`

##### 探索属性
- `travel_minutes`
- `threat_level`
- `discoverable`
- `recommended_auxiliary_tools`

##### 掉落相关
- `loot_bias_tags`
- `ai_exploration_rule`（只供 AI 结算/叙事使用，不直接显示给玩家）

##### 解锁相关
- 解锁关系不写在地点资源中，而由 `levels/outing/unlock_links/*.tres` 描述
- 解锁条件包含 `from_location_id`、`to_location_id`、`required_success_count`、`unlock_key`

##### 表现相关
- `detail_image`
- `region_tint`
- `marker_variant`

### 9.2 掉落规则

当前 MVP 使用标签驱动掉落：地点配置 `loot_bias_tags`，控制器中的 `LOOT_ITEM_PATHS` 将标签映射到可获得物品。

后续如需更精细控制，可再拆出独立掉落资源；但当前规则必须保持：

- 地点只给“偏向标签”
- 控制器根据标签和随机数生成 2~5 次物资结果
- 工具/特殊携带物可提高搜索收益，但不直接写死某个地点必出某物

### 9.3 地图状态保存

需要一份运行时或存档数据记录：

- 已解锁地点集合：`unlocked_location_ids`
- 已发现解锁 key：`discovered_unlock_keys`
- 各地点成功探索次数：`successful_explore_counts`

地图解锁状态不应硬写回地点模板资源，而应保存在 `/root/Global` 的 `outing_map_progress_runtime` 中，并由 `SaveManager` 通过 `Global.build_global_save_payload()` 写入存档。

## 10. 物资来源规则

本设计已更新：

**外出携带物资从庇护所总库存汇总中读取，而不是只读角色背包。**

### 选择该方案的原因

- 当前项目的核心物资管理发生在 3D 柜子内；食品柜、医疗柜、武器/工具柜、材料箱都是真实库存来源。
- 外出前玩家不应逐个柜子查找物资，准备面板应提供汇总视图。
- 汇总视图只负责“选择”，真实扣除仍回到来源柜子的具体 slot。
- 武器、工具、特殊物品通常不需要 3D 模型摆放在角色身上，作为数据库存即可。

因此：

- `ShelterInventoryResource` 汇总所有 `include_in_outing_pool = true` 的库存源。
- 每个外出条目记录 `source_id`、`source_name`、`slot_index`、`item`、`amount`。
- 确认外出时，系统按 `source_id:slot_index` 从真实来源扣除。
- 武器/工具/特殊物品外出后归还原来源格；食品、医疗、材料按消耗品处理。

---

## 11. 携带栏规则

当前实现采用 **12 格外出携带栏**，而不是容量值。

### 基本规则

- 携带栏固定 12 格。
- 一个物品占一格。
- 非武器同来源、同物品可在同一格内堆叠，数量不超过 `ItemData.MaxStackSize`。
- `outing_category == "weapon"` 的物品永远不堆叠，一件占一格。
- 已选择数量不能超过来源柜子当前库存数量。

### UI 要求

准备面板中需要实时显示：

- 已用格数 / 12
- 已选择物品列表
- 来源柜名称
- 非武器堆叠数量
- 武器独立占格

---

## 12. 时间推进规则

第一版外出唯一明确成本是 **时间**。

### 总耗时公式

`total_minutes = round_trip_minutes + search_minutes`

含义：
- 往返路程：距离越远，回来时间越长
- 搜索时间：由地点基础时长与威胁等级决定

后续可以加天气、事故、工具不足等修正，但第一版不加入这些因素。

---

## 13. 探索结算规则

一次探索结算包含以下部分：

1. 扣除/归还携带物资
2. 抽取物资结果
3. 写入庇护所库存
4. 记录成功探索次数并解锁地点
5. 输出结构化结果数据
6. 触发自动保存

### 13.1 掉落逻辑

每个地点拥有 `loot_bias_tags`，通过标签映射到候选物资。

#### 基本原则
- 掉落是随机的
- 地点本身有高概率掉落倾向
- 玩家不是精确指定产出，而是进入一个“有倾向的随机结果”

### 13.2 掉落次数

当前控制器按 2~5 次抽取。威胁较高、携带工具/特殊物品时可提高抽取次数。

### 13.3 权重修正

本次已确认：
- 第一版不做机器人损坏
- 不做电量
- 不做额外状态惩罚

因此第一版掉落权重主要受以下因素影响：

- 地点原始权重
- 地点偏向标签
- 随机结果

第一版先不加入复杂偏好系统和工具条件链。后续如有需要，再增加玩家“搜索偏向”选项。

### 13.4 结果风格

结果应当：
- 具有明显地点特色
- 保留随机波动
- 不保证绝对产出

---

## 14. 新地点解锁规则

当前采用 `unlock_links/*.tres` 的显式连线规则：

- 每次成功探索调用 `record_success(location_id)`
- 遍历从当前地点出发的解锁链接
- 当 `successful_explore_counts[from_location_id] >= required_success_count` 时解锁 `to_location_id`

### 设计目的

- 让地图逐步向外扩展
- 让“更远地点 = 更晚触达”成立
- 避免第一版复杂条件树

更复杂的发现率、道具影响、探索等级等逻辑留待后续扩展。

## 15. 结果数据结构

建议探索结算后生成统一结果对象，供 UI 展示、存档记录、后续 AI 叙事系统使用。

建议字段：

- `rule` / `location_id` / `location_name`
- `commit`：携带、归还、消耗统计
- `loot`：生成的带回物资
- `deposit`：成功入库数量与丢失数量
- `unlocked`：本次发现的新地点
- `risk`：基于威胁和携带物资的结构化风险判断
- `time`：路程、搜索、总耗时

### 原则

- 规则系统负责生成结构化结果
- AI 系统以后只负责把结构化结果转成叙事文本

这可以避免把 AI 叙事绑死到核心结算逻辑里。

## 16. 存档与持久化

外出系统相关状态必须进入全局存档，不能依赖 `.tres` 模板被运行时改写。

### 16.1 全局存档入口

`SaveManager` 通过以下接口采集全局状态：

- `Global.build_global_save_payload()`
- `Global.apply_global_save_payload(payload)`

### 16.2 必须保存的状态

- `outing_return_scene_path`：进入外出地图前的场景路径，保证返回正确场景。
- `shelter_inventory`：所有庇护所库存源的 slot 数据。
- `outing_map_progress`：地图发现/探索进度。

### 16.3 外出结算后的保存时机

确认外出结算后，只要发生以下任一变化，就必须触发保存：

- 携带物资被扣除或归还
- 搜索获得物资写入柜子或外出带回包
- 新地点解锁
- 成功探索次数变化

返回避难所成功后，也要再保存一次，确保 `outing_return_scene_path` 被清空。

### 16.4 相关实现文件

- `scripts/global.gd`：运行时全局库存与地图进度、全局 payload。
- `scripts/system/save_manager.gd`：保存/读取 `global_data`。
- `scripts/Inventory/shelter_inventory_resource.gd`：外出携带汇总、扣除、归还、入库。
- `levels/outing/resources/outing_map_progress_resource.gd`：地图发现进度资源。
- `levels/outing/outing_map_level_v3.gd`：外出结算、入库、地图解锁、保存触发。

第一版结果面板显示：

- 去了哪里
- 花了多久

## 18. 风格方向

整体风格参考“地图探索 UI”，但不直接照搬参考图。

### 风格关键词
- 旧地图感
- 纸张 / 档案夹板感
- 暖灰 / 褐色 / 暗绿 / 米白
- 地点高亮使用偏金黄 / 浅橙
- 以地图本体为主，不做过重科技风

### 设计约束
- 地图是主视觉
- 卡片只承担说明与确认
- 准备面板结构清晰，不复刻复杂背包大 UI

## 19. MVP 实现范围

### 包含
- 独立 2D 外出地图场景
- 地图拖拽与缩放
- 地点点位显示与点击
- 地点详情卡
- 从庇护所汇总库存选择携带物资
- 12 格携带栏限制

- 距离与搜索耗时展示
- 外出获得物资写回庇护所库存

## 20. 后续扩展预留

该设计为后续扩展保留以下接口方向：

- AI 探索叙事生成
- 更复杂的地点发现条件
- 偏好搜索模式
- 外出事件系统
- 工具标签影响掉落
- 电量/损坏系统
- 地图区域层级与更远区域探索

---

## 21. 结论

第一版外出地图系统应优先打通如下闭环：

**进入地图 → 点击地点 → 查看详情 → 从庇护所汇总库存选择携带物资 → 出发 → 扣除/归还携带物 → 获得物资并写回库存 → 解锁新地点并保存 → 返回避难所**

这条链稳定后，再逐步叠加偏好、事件、AI 叙事和更复杂的外出成本系统。
